# 人間テストシナリオ v0.9.0 (QA-01)

最終更新: 2026-02-14 (JST)

## 1. 目的

Runbook のみで障害対応と監査記録が完遂できるかを、
実運用に近い条件で検証する。

## 2. 体制

- 進行役: 1名
- 実行者: 2名（一次対応 / 記録担当）
- 観察者: 1名（詰まり・曖昧表現の記録）

## 3. 事前準備

- [ ] 対象Runbookを事前配布
- [ ] 演習環境を固定（`stg` 推奨）
- [ ] 連絡経路（Incident チャンネル）確認
- [ ] 記録テンプレート準備

## 4. シナリオ定義

### QA-01-1 更新失敗（RNB-01）

- 入力条件: 更新ジョブ失敗ログを提示
- 期待行動: 再実行可否と分岐先を判断
- 合格条件: 30分以内に復旧方針決定
- 参照: `docs/RUNBOOK_UPDATE_FAILURE_v0_9_0.md`

### QA-01-2 キャッシュ障害（RNB-02）

- 入力条件: stale 結果と Redis エラーログを提示
- 期待行動: 接続失敗/stale/miss暴走を切り分け
- 合格条件: 25分以内に緩和策実行
- 参照: `docs/RUNBOOK_CACHE_INCIDENT_v0_9_0.md`

### QA-01-3 DBロールバック判断（RNB-03）

- 入力条件: データ不整合と5xx悪化を提示
- 期待行動: ロールバック可否と復元ポイントを判断
- 合格条件: 40分以内に実施判断と復旧判定案を提示
- 参照: `docs/RUNBOOK_DB_ROLLBACK_v0_9_0.md`

### QA-01-4 月次監査レポート作成（AUD-01）

- 入力条件: 更新履歴・指標値・障害記録を提供
- 期待行動: テンプレートへ記録し第三者追跡可能にする
- 合格条件: 30分以内にドラフト完成
- 参照: `docs/MONTHLY_AUDIT_REPORT_TEMPLATE_v0_9_0.md`

## 5. 判定基準

- `Pass`: 手順逸脱なし、証跡が十分
- `Conditional Pass`: 復旧可能だがRunbook改善が必要
- `Fail`: 復旧不能または判断不能

`Fail` が1件でも出た場合、`QA-03` でRunbook改訂を先行する。
